# 国家标准设备管理系统 (EMS) - 需求分析报告

## 1. 引言

### 1.1 编写目的
明确“国家标准设备管理系统”的业务、功能及数据需求，包含原有的设备、分类、单位、调拨、检修、报废模块，以及新引入的 **登录鉴权与 RBAC (基于角色的权限控制) 体系**，为系统开发及测试提供最新基准。

### 1.2 项目背景
随着企事业单位固定资产规模的扩大，传统的纸质或简单表格管理已无法满足设备全生命周期监管需求。本项目旨在开发一套符合 GB/T 14885 标准设备分类体系、支持多角色协作、具备高安全性的自动化管理系统。

---

## 2. 业务需求概述

系统主要服务于企事业单位的设备使用者、技术维护人员、资产管理员及运维人员，核心目标包括：
*   **规范化分类**：严格遵循国家标准设备分类编码（GB/T 14885）。
*   **全生命周期追踪**：记录设备从入库、在用、维修到报废的每一个状态变更。
*   **动态调拨管理**：实现设备在不同部门间的透明流转。
*   **数据安全备份**：提供一键 SQL 备份与恢复。
*   **四级角色鉴权 (新)**：系统分为设备操作员、维修工程师、资产管理员、系统管理员四级，实现菜单级与接口级权限拦截。

## 3. 功能需求分析

### 3.1 账户鉴权与权限管理 (新)
*   **用户登录**：支持输入用户名及密码登录，后端基于 MD5 加密比对，并下发有效期 12 小时的 JWT Token。
*   **自助注册**：供普通操作员自助注册。注册新用户时，角色一律强制默认为 `0` (设备操作员)，用户名全局唯一。此外，注册表单必须收集 `realName` 与有效的所属部门/单位 (`unitCode`)，并在注册时由后端校验单位存在性。
*   **用户权限管理**：仅系统管理员（`role=3`）可见，支持查看所有用户账户列表（通过专用的 `UserVO` 进行敏感信息如密码哈希的脱敏），并在管理后台执行以下操作：
    - **资料编辑**：支持同步修改用户的真实姓名 (`realName`)、系统角色 (`role`) 和所属单位 (`unitCode`)，用户名 `username` 作为稳定登录标识不允许修改。
    - **管理员联动清除逻辑**：当将用户的角色修改为“系统管理员”时，其所属单位将被联动清空为 `NULL`（代表全局角色），且前端单位选择框自动置灰禁用。其余业务角色（操作员、维修工程师、资产管理员）则强制要求绑定有效的所属单位。
    - **设备保管强阻断校验**：当管理员更改用户的所属单位（包括因角色变更导致单位改变或置空）时，如果该用户当前名下仍有正在保管的设备（即 `custodian` 指向该用户的设备数量大于 0），后端会利用数据库事务强制回滚更新并抛出业务异常，拦截部门变动，要求先清退或移交设备。
    - **管理员重置密码**：系统管理员可以为其他账号直接设置新密码，用于忘记密码或初始化场景，但不能通过该入口重置自己的密码。
    - **账号删除与级联清退**：支持删除停用账号。删除时若用户名下仍有保管设备，系统自动清空设备保管人并写入“自动退还”流水；若用户仍有关联未完结检修工单，则后端阻断删除。
*   **用户自助改密**：所有已登录用户均可通过右上角用户区域的“修改密码”入口完成本人密码修改。流程要求输入旧密码、新密码、确认新密码；修改成功后系统清空当前登录态并要求重新登录。

### 3.2 基础数据管理
*   **分类管理**：维护国标分类编码、预计使用年限及残值率。
*   **单位管理**：维护设备使用部门及负责人信息。

### 3.3 设备业务管理
*   **设备入库**：登记设备编号、名称、原值、购入日期等基础信息。当前代码中的设备状态枚举为“在用 / 维修 / 报废”，未单独落库“闲置”状态；可领用设备以 `status = '在用'` 且 `custodian IS NULL` 表示“可分配/无保管人”。
*   **设备领用与审批工作流 (新)**：
    - **领用申请**：普通用户或操作员可对本单位内 `status = '在用'` 且 `custodian IS NULL` 的无保管人设备发起领用申请。发起后设备仍保持“在用”业务状态，但由 `t_equipment_claim.status = 0` 的待审批流水锁定，防止重复申请。
    - **审批权限与隔离**：只有设备当前所属单位的“资产管理员”（`role=2`）才能审批该设备的领用申请。非该单位管理员无法查看及审批，阻止跨单位越权。
    - **审批同意**：管理员审批通过后，领用记录状态更新为“1-已同意”，设备保管人（`custodian`）被赋值为申请人用户名，设备状态同步变更为“在用”。
    - **审批拒绝**：管理员审批拒绝时，领用记录状态更新为“2-已拒绝”，设备保持“在用”且保管人仍为 `NULL`，回到可被重新申请或直接分配的状态。
    - **设备退还**：用户可在个人中心或设备台账对正保管的设备发起“退还”操作。退还后设备保持“在用”，保管人清空为 `NULL`，领用记录表中插入一条状态为“4-已退还”的流水记录。
    - **直接分配 (辅助)**：资产管理员可以直接在后台为指定操作员分配设备（无需经历申请审批流程），直接写入领用流转流水，设备状态变为“在用”。
    - **删除用户联动清退**：当管理员删除某一用户时，后端在事务中自动将其名下保管的所有设备重置为无保管人（`custodian = NULL`），设备保持“在用”，并生成一条“用户删除自动退还”的流转记录；若用户仍有关联未完结检修工单，则阻断删除。
*   **设备调拨**：支持在不同单位之间调拨，记录调拨日期、变动类型、原单位及新单位信息。
*   **检修维护**：操作员可对正保管的设备发起故障申报，资产管理员将其指派给本单位的维修工程师（`role=1`）。检修流转包含待指派（0）、维修中（1）、待复核（2）、已复核可用（3）、已复核转报废（4）状态。只有被指派的维修工程师或本单位资产管理员才能操作完工（工单状态置为 2，且此时设备依然保持“维修”状态）。随后，需由本单位资产管理员执行“完工复核”，决定是“复核通过恢复可用”（工单置 3，设备状态恢复为在用）或“复核转报废”（工单置 4，设备状态变更为报废，清空保管人，插入报废记录，并关联退领流水）。
*   **报废处理**：对折旧期满或损坏无法维修的设备，由资产管理员执行报废登记，记录报废原因与审批人，设备状态变更为“报废”。已报废的设备不允许执行调拨、领用或检修操作。

### 3.4 系统安全与维护
*   **数据备份**：调用 `mysqldump` 生成 SQL 脚本并按时间戳命名存储。
*   **数据恢复**：支持选择已有的备份文件进行一键还原。

### 3.5 业务数据看板 (新)
*   **登录后默认首屏**：提供基于 RBAC 角色的定制化数据看板首屏，支持各角色核心指标卡片、图表和待办面板展示。
*   **资产管理员看板 (Role 2)**：
    - 展示全局资产总数、总资产、在用、维修、报废资产状态分布。
    - 提供分类资产占比图、部门资产金额分布图及全局维保费用趋势图。
    - 聚合展示本部门下“待审批的领用申请”和“待指派的维修单”，提供快速流转入口。
*   **系统管理员看板 (Role 3)**：
    - 展示只读全局设备总数、总原值及设备状态分布，不提供业务流转入口。
    - 提供系统内用户角色分布图表。
    - 聚合展示最近生成的 SQL 数据备份列表、文件大小和状态。
*   **维修工程师看板 (Role 1)**：
    - 展示分配给当前维修工的未完成工单列表和维修中工单列表。
    - 展示工单完成率及个人的维保完成与维修费用趋势图。
*   **设备操作员看板 (Role 0)**：
    - 展示当前用户保管的设备卡片及状态。
    - 聚合本人在途的领用申请列表和在途的报修进度列表。
    - 展示名下资产折旧净值统计。

### 3.6 数据治理与运营风险分析 (新)
*   **治理目标**：在数据看板和操作审计稳定后，为资产管理员和系统管理员提供可解释的数据质量检查、风险设备识别、维修成本异常分析和资产健康评分。
*   **数据质量检查**：
    - 识别设备状态非法或缺失、设备分类/单位/购入日期/原值缺失或异常。
    - 识别设备保管人与所属单位不一致、维修记录关联设备不存在或维修人员缺失。
    - 识别疑似重复设备，第一版按 `equipName + model + unitCode + purchaseDate + originalValue` 组合判断。
*   **风险设备识别**：
    - 年限风险按 `已使用月数 / (预计使用年限 * 12)` 计算。
    - 高风险设备：未报废设备满足使用年限占比 `>= 0.9`、维修次数 `>= 3`、维修费用占原值 `>= 0.3` 任一条件。
    - 中风险设备：未报废设备满足使用年限占比 `>= 0.75`、维修次数 `>= 2`、维修费用占原值 `>= 0.15`、当前状态为“维修”任一条件。
    - 低风险设备：未命中中高风险的在册设备。
*   **运营分析范围**：
    - 高频维修设备 Top N。
    - 维修费用占原值异常设备。
    - 长期空闲设备，第一版定义为 `custodian IS NULL` 且 `status = '在用'`。
    - 部门维度和分类维度的风险分布。
*   **角色权限**：
    - Role 2 资产管理员仅可查看本单位或业务管理范围内的数据治理结果。
    - Role 3 系统管理员可全局只读查看治理结果。
    - Role 0 设备操作员和 Role 1 维修工程师不可访问数据治理功能。
*   **边界说明**：第一版不引入 AI、不新增依赖、不修改数据库结构、不做规则后台配置化。

### 3.7 事找人 MVP 与消息中心 (新)
*   **能力定位**：在数据治理、领用审批和检修闭环稳定后，将确定性异常和待办升级为“规则事件”，由系统主动路由给责任人。
*   **首批事件类型**：
    - 高风险设备提醒：当设备命中高风险规则时，通知该设备所属单位的资产管理员。
    - 领用审批积压提醒：当某单位待审批领用记录数或等待时长超过阈值时，通知对应单位资产管理员。
    - 维修超时提醒：当维修工单在“维修中”状态停留超过阈值时，通知当前维修工程师，必要时抄送本单位资产管理员。
*   **消息中心能力**：
    - 提供事件列表、未读数、已读状态和事件详情。
    - 事件可跳转到治理页、设备详情页、领用审批页或检修页继续处理。
*   **角色权限**：
    - Role 2 资产管理员可查看分配给本人的风险和审批类事件。
    - Role 1 维修工程师可查看分配给本人的维修超时事件。
    - Role 3 系统管理员仅保留全局只读视角，不承担日常事件处理责任。
    - Role 0 设备操作员不接收该 MVP 范围内的管理类事件。
*   **边界说明**：第一版不做 WebSocket、不做企业微信/邮件/短信对接、不做 AI 自动生成事件。

### 3.8 AI 辅助报告与建议草案 (新)
*   **AI 辅助定位**：在数据看板、操作审计、生命周期详情和数据治理能力稳定后，为资产管理员和系统管理员提供自然语言报告、异常解释、设备生命周期摘要和处置建议草案。
*   **资产运营报告草案**：
    - 支持生成周报或月报草案。
    - 报告内容包括资产概况、风险摘要、维修成本异常、待办积压、数据质量问题和建议动作。
    - 报告仅作为人工编辑草案，不自动提交、不自动归档。
*   **设备生命周期摘要**：
    - 基于单台设备的基础信息、折旧、领用、维修、调拨、报废和审计时间线生成摘要。
    - 输出主要风险证据和人工复核建议。
*   **角色权限**：
    - Role 2 资产管理员可在本单位业务范围内生成报告和设备摘要。
    - Role 3 系统管理员可全局只读生成审计型报告。
    - Role 0 设备操作员和 Role 1 维修工程师不可访问 AI 辅助功能。
*   **安全边界**：
    - AI 不直接访问数据库，只接收后端已授权、已裁剪的 DTO/VO 上下文。
    - AI 不调用审批、报废、调拨、删除、数据库恢复等业务写接口。
    - 未配置 AI Provider/API Key 或 Provider 调用失败时，应返回可诊断错误，不影响非 AI 业务功能。
*   **边界说明**：第一版不实现自由 SQL 问答、RAG 知识库、AI 自动决策或 AI 输出持久化。

| 字段名 | 数据类型 | 长度 | 约束 | 备注/说明 |
| :--- | :--- | :--- | :--- | :--- |

| `password` | VARCHAR | 100 | NOT NULL | 登录密码 (MD5密文) |

## 5. 非功能需求
*   **安全性**：
    - 对数据恢复等高危操作设置强提醒。
    - 接口强制进行 JWT 签名和有效性鉴权拦截，阻止未登录与越权调用。
    - **数据隔离与权限隔离**：基于单位进行水平隔离，不同单位的用户只能查看/操作本单位的设备及相关单据记录。对于核心操作（指派工单、领用审批）实施基于角色的垂直与水平越权校验，拦截一切非法调用。
*   **健壮性**：后端数据库事务应保证调拨、领用审批、检修及报废的多表写一致性，数据备份还原过程具备良好的异常阻断，若发生业务逻辑冲突（如保管设备未清退进行单位调整）保证事务一致性物理回滚。
*   **易用性**：前端依据登录用户的角色展示动态侧边栏，页面级元素及按钮（如“审批”、“修改单位”）根据所属权限与联动逻辑规则进行动态显隐、启用或置灰禁用。

## 6. 当前代码对齐说明
- **认证边界**：后端仅放行 `/users/login`、`/users/register`、跨域预检 `OPTIONS` 和注册页需要读取的 `GET /departments`；其余接口要求请求头携带 `token`。
- **单位隔离**：`LoginCheckInterceptor` 解析 JWT 后会实时从数据库读取用户最新 `unitCode` 写入 `BaseContext`，防止用户单位变更后旧 Token 保留旧单位边界。
- **设备状态口径**：当前代码只使用“在用 / 维修 / 报废”三类设备状态。“可领用 / 可分配”不是独立状态，而是 `custodian IS NULL` 的派生业务条件。
- **消息同步时机**：消息中心接口按当前登录用户同步并查询消息，事件来自治理高风险设备、待审批领用单和维修中超时工单，不由 AI 自动生成。
- **AI 安全口径**：AI 请求由后端组装裁剪后的看板、治理和设备详情上下文；未配置 API Key 时返回可诊断错误，不影响非 AI 模块。
