# 国家标准设备管理系统 (EMS) - 系统架构设计 (architecture.md)

## 1. 业务分层与技术架构
本系统后端严格采用三层分层开发架构，各层之间职责分明，杜绝越权或跳跃调用：
```
Web 浏览器 (SPA) ➔ 控制层 (Controller) ➔ 业务逻辑层 (Service) ➔ 数据访问层 (DAO) ➔ 物理数据库
```
*   **控制层 (Controller)**：仅负责 HTTP 请求的分发、数据包装（统一 Result 返回）与请求入参的基本格式校验，严禁写入业务逻辑。
*   **业务逻辑层 (Service)**：系统核心业务在此实现，负责处理状态流转、计算公式及多表关联写事务控制。
*   **数据访问层 (DAO)**：基于原生 SQL 和 `JdbcTemplate` 与数据库交互，不写业务逻辑。

## 2. 概念结构设计 (E-R拓扑模型)
通过对系统内实体、属性及其联系进行抽象，建立独立于具体关系数据库的概念模型。

### 2.1 局部 E-R 关系描述
1.  **设备基础与分类**：
    *   “设备”与“设备分类”属于多对一联系（`1:n`）。一个分类包含多台设备，一台设备仅属于一个分类。
2.  **设备调拨与单位**：
    *   “设备”与“使用单位”属于多对多联系（`m:n`）。联系自身记录调拨日期、变动类型、经办人及调拨原因。
3.  **设备检修维保**：
    *   “设备”与“检修单”属于一对多联系（`1:n`）。一台设备可对应多条检修记录。
4.  **设备报废处置**：
    *   “设备”与“报废单”属于一对一联系（`1:1`）。一台设备仅对应一条唯一的报废记录。
5.  **用户与权限控制 (新)**：
    *   “用户”实体与“设备”实体通过“保管”关联（`1:n`），即一个用户能保管多台设备，每台设备只能有唯一保管人。
    *   “用户”实体与“检修单”实体通过“报修”关联（`1:n`），即一个用户可提交多次报修。

### 2.2 全局 E-R 实体属性映射
*   **设备 (Equipment)**：[设备编号, 设备名称, 规格型号, 原值, 购入日期, 状态, 保管人]
*   **使用单位 (Department)**：[单位代码, 单位名称, 负责人]
*   **国家标准分类 (Category)**：[分类编码, 分类名称, 预计使用年限, 残值率]
*   **系统用户 (User)**：[用户名, 密码, 真实姓名, 角色]
*   **检修记录 (Maintenance)**：[检修单号, 检修日期, 检修内容, 维修费用, 检修人, 报修人, 工单状态]
*   **报废登记 (Scrap)**：[设备编号, 报废单号, 报废日期, 审批人, 原因]
*   **调拨记录 (Transfer)**：[调拨单号, 调拨日期, 变动类型, 原单位, 新单位, 经办人, 调拨说明]
