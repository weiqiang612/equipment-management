# 国家标准设备管理系统 (EMS) - 系统架构设计 (architecture.md)

## 1. 业务分层与技术架构
本系统后端严格采用三层分层开发架构，各层之间职责分明，杜绝越权或跳跃调用：
```
Web 浏览器 (SPA) ➔ 控制层 (Controller) ➔ 业务逻辑层 (Service) ➔ 数据访问层 (DAO) ➔ 物理数据库
```
*   **控制层 (Controller)**：仅负责 HTTP 请求的分发、数据包装（统一 Result 返回）与请求入参的基本格式校验，严禁写入业务逻辑。
*   **业务逻辑层 (Service)**：系统核心业务在此实现，负责处理状态流转、计算公式及多表关联写事务控制。
*   **数据访问层 (DAO)**：基于原生 SQL 和 `JdbcTemplate` 与数据库交互，不写业务逻辑。

## 2. 演进架构规划

系统后续演进在现有三层架构上增加四类横向能力，所有能力仍遵守 Controller -> Service -> DAO 的调用边界。

### 2.1 看板聚合层
*   **定位**：为数据看板提供跨模块聚合统计，避免前端拉取全量列表后自行统计。
*   **实现边界**：新增 Dashboard Controller/Service/DAO 和 Dashboard VO，使用 `COUNT`、`SUM`、`GROUP BY` 等数据库聚合能力。
*   **权限要求**：聚合结果必须基于 `BaseContext` 中的角色、用户名和单位代码进行裁剪。

### 2.2 操作审计层
*   **定位**：记录关键业务动作，为系统管理员审计、设备生命周期追溯和后续 AI 报告提供可信上下文。
*   **实现边界**：新增追加式 `operation_log` 表，通过 Service 层在关键业务事务中写入审计记录。
*   **安全要求**：审计日志默认只追加，不提供编辑或删除能力；Role 3 只读查看全局审计。

### 2.3 生命周期聚合层
*   **定位**：以设备为主线聚合分散在设备、领用、维修、调拨、报废和审计模块中的历史记录。
*   **实现边界**：新增设备详情聚合接口，返回 DTO/VO，前端以时间线和分区卡片展示。
*   **权限要求**：Role 0/1 只能查看与本人相关的数据，Role 2/3 按既有业务和审计边界查看。

### 2.4 数据治理与事件提醒层
*   **定位**：在看板和审计稳定后，先基于确定性规则补齐数据质量检查、风险识别和“事找人”事件提醒能力。
*   **数据治理边界**：第一版通过 `GovernanceController -> GovernanceService -> GovernanceDao` 提供实时聚合和分页查询，不新增数据库表、索引或迁移脚本。
*   **规则计算方式**：风险阈值以 Service 层命名常量实现，包括使用年限占比、维修次数、维修费用占原值和当前状态；数据质量问题作为查询结果降级返回，不导致接口 500。
*   **权限要求**：Role 2 按单位边界查看治理结果，Role 3 全局只读，Role 0/1 禁止访问治理接口。
*   **事件提醒边界**：第一版事件中心只覆盖高风险设备、审批积压和维修超时三类事件；事件生成依赖规则与现有业务状态，不依赖 AI 判定。
*   **路由方式**：事件按角色、所属单位、工单归属等确定性条件路由到责任人；消息中心采用拉取式刷新，不依赖 WebSocket。

### 2.5 AI 辅助层
*   **定位**：在治理和事件提醒稳定后，再增加 AI 报告、解释和建议能力。
*   **AI 边界**：先使用可解释规则生成候选风险，再由 AI 进行总结、解释和建议草案生成。
*   **AI 编排方式**：AI 辅助能力通过 `AiAssistantController -> AiAssistantService -> AI Provider Adapter` 调用外部模型；上下文由后端从 Dashboard、Governance、Operation Log 和 Equipment Detail 的 DTO/VO 摘要中组装。
*   **Provider 降级**：AI Provider 未配置、超时或失败时，服务返回明确错误，不影响看板、治理、审计和设备详情等非 AI 功能。
*   **数据最小化**：发送给 AI 的上下文必须过滤密码、Token、完整用户敏感信息和无关全量业务数据。
*   **安全要求**：AI 不直接调用高风险业务写接口，所有审批、报废、恢复数据库等动作必须由人工确认并审计留痕。

## 3. 概念结构设计 (E-R拓扑模型)
通过对系统内实体、属性及其联系进行抽象，建立独立于具体关系数据库的概念模型。

### 3.1 局部 E-R 关系描述
1.  **设备基础与分类**：
    *   “设备”与“设备分类”属于多对一联系（`1:n`）。一个分类包含多台设备，一台设备仅属于一个分类。
2.  **设备调拨与单位**：
    *   “设备”与“使用单位”属于多对多联系（`m:n`）。联系自身记录调拨日期、变动类型、经办人及调拨原因。
3.  **设备检修维保**：
    *   “设备”与“检修单”属于一对多联系（`1:n`）。一台设备可对应多条检修记录。
4.  **设备报废处置**：
    *   “设备”与“报废单”属于一对一联系（`1:1`）。一台设备仅对应一条唯一的报废记录。
5.  **用户与权限控制 (新)**：
    *   “用户”实体与“设备”实体通过“保管”关联（`1:n`），即一个用户能保管多台设备，每台设备只能有唯一保管人。
    *   “用户”实体与“检修单”实体通过“报修”关联（`1:n`），即一个用户可提交多次报修。
6.  **设备领用与审批 (新)**：
    *   “用户”与“设备”通过“领用申请单”产生多对多联系（`m:n`），即一个用户可对多台设备发起多笔领用申请。该联系转化为物理表后记录申请流水信息，并在审批成功后联动修改“设备”与“用户”的保管关联关系。

### 3.2 全局 E-R 实体属性映射
*   **设备 (Equipment)**：[设备编号, 设备名称, 规格型号, 原值, 购入日期, 状态, 保管人]
*   **使用单位 (Department)**：[单位代码, 单位名称, 负责人]
*   **国家标准分类 (Category)**：[分类编码, 分类名称, 预计使用年限, 残值率]
*   **系统用户 (User)**：[用户名, 密码, 真实姓名, 角色, 所属单位]
*   **检修记录 (Maintenance)**：[检修单号, 检修日期, 检修内容, 维修费用, 检修人, 报修人, 工单状态]
*   **报废登记 (Scrap)**：[设备编号, 报废单号, 报废日期, 审批人, 原因]
*   **调拨记录 (Transfer)**：[调拨单号, 调拨日期, 变动类型, 原单位, 新单位, 经办人, 调拨说明]
*   **设备领用申请 (EquipmentClaim)**：[领用申请单号, 设备编号, 申请人, 审批人, 领用状态, 备注, 创建时间, 更新时间]

## 4. 安全鉴权与 RBAC 防御架构设计

本系统构建了基于“前端单页拦截 + 后端双层硬过滤”的多维度权限防御体系，以杜绝一切水平和垂直越权操作。

### 4.1 后端双层鉴权拦截机制
后端安全框架主要利用 Spring MVC 拦截器和 AOP 切面注解进行协同工作：

```
客户端请求 ➔ 1. LoginCheckInterceptor (JWT 认证) ➔ 2. BaseContext (线程上下文) ➔ 3. AOP 切面 (@RequiresRoles 鉴权) ➔ 控制器 (Controller)
```

1.  **第一层：JWT 登录状态鉴权拦截器 (`LoginCheckInterceptor`)**
    *   拦截器在 Spring MVC 网关层拦截所有非白名单请求（如 `/users/login` 和 `/users/register` 自动放行）。
    -   从 HTTP Header `token` 中提取 JWT，调用 `JwtUtils.parseJWT()` 进行解析。
    -   若 Token 校验成功，将解析出的用户核心元数据（用户 ID `id`、用户名 `username`、角色代号 `role`、部门代码 `unitCode`）绑定到基于线程变量 `ThreadLocal` 的上下文容器 `BaseContext` 中，供后续逻辑调用。
    -   若校验失败（如未登录、Token 过期），拦截器直接拦截并返回统一响应（Result.code=0，msg='未登录'），响应 HTTP 状态码为 401。
2.  **第二层：基于 AOP 切面的注解级越权硬拦截**
    -   系统定义了自定义注解 `@RequiresRoles`（可加于类级别或方法级别），用于指定访问当前资源所需具备的最低角色代号（如 `2, 3` 代表管理员与系统管理员可写）。
    -   利用 AspectJ 切面拦截含有该注解的方法。切面通知在方法执行前，读取 `BaseContext` 中绑定的当前登录用户的 `role` 值。
    -   如果当前用户的 `role` 不匹配注解所要求的合法范围，切面立即中断执行，向调用栈抛出自定义的越权业务异常 `ForbiddenException`。
3.  **第三层：全局越权异常统一捕获 (`GlobalExceptionHandler`)**
    -   在全局异常处理器中，通过 `@ExceptionHandler(ForbiddenException.class)` 对抛出的越权异常进行物理捕获。
    -   处理器强制设置 HTTP 响应状态码为 **403 (Forbidden)**，并返回脱敏的统一报文 `{ "code": 0, "msg": "权限不足，拒绝访问", "data": null }`，防止了恶意攻击者刺探接口信息。

### 4.2 前端单页应用 (SPA) 鉴权机制
1.  **路由守卫与白名单拦截**：在 `router/index.js` 中使用全局守卫 `router.beforeEach`，根据目标路由的 `meta.roles` 配置，检查当前 LocalStorage 中缓存用户的 `role`。若用户试图越权进入，强制拦截并重定向回主页或错误页。
2.  **细粒度前端操作隔离**：通过 Vue 指令或条件渲染（`v-if`），结合当前用户的 `role` 与单据本身的归属字段（如 `custodian`），实现“设备台账”与“检修记录”操作按钮的动态显隐和只读切换。
