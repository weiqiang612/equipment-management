# 国家标准设备管理系统 (EMS) - 原型与 UI 设计规范 (ui_prototype.md)

## 1. 主视觉与色彩规范
本系统采用 **科技工业蓝灰** 视觉体系，界面要求清晰、现代且符合国家标准软件的使用习惯。
*   **主色 (Primary)**：`#409EFF` (Element UI 经典蓝)
*   **成功色 (Success)**：`#67C23A` (用于“在用”设备状态、备份成功标识)
*   **警告色 (Warning)**：`#E6A23C` (用于“维修”设备状态、恢复确认提示)
*   **危险色 (Danger)**：`#F56C6C` (用于“报废”设备状态、删除设备、清除 Token 退出)
*   **侧边栏背景**：`#304156` (科技暗蓝灰)

---

## 2. 界面布局结构
系统采用经典的后台管理左右分栏布局：
1.  **左侧导航栏 (Aside)**：
    *   折叠/展开宽度：200px。
    *   **RBAC 动态隐藏**：依据当前用户的 `role` 渲染菜单。操作员只可见“我的设备”、“报修申请”；其他角色可见“设备资产管理”、“分类管理”、“单位管理”等。
2.  **右侧主区域**：
    *   头部 (Header)：面包屑导航、用户真实姓名欢迎词及当前角色 Tag 标签、“修改密码”入口与退出登录按钮。
    *   主视图 (Main)：背景色 `#f0f2f5`，留白 20px，内部使用卡片组件 (`el-card`) 承载表格和检索表单。

## 4. UI 组件像素级还原准则
*   **组件命名**：Vue 组件文件命名一律使用大驼峰且不使用单个单词（如 `UserManage.vue` 而非 `User.vue`）。
*   **自适应设计**：表格和表单必须支持在 1280px 至 1920px 宽度下的自适应缩放，超出部分使用分页器拦截，禁止页面底端出现横向滚动条。
*   **操作反馈**：
    *   删除、恢复、退出等敏感操作必须使用 `this.$confirm` 弹出危险提示确认框。
    *   接口交互成功后必须触发 `this.$message.success` 进行吐司提示。

### 4.1 用户权限管理页补充规范
*   页面标题保持“用户权限管理”，仅系统管理员可从左侧导航进入。
*   表格至少展示：用户名、真实姓名、系统角色、所属单位、创建时间、更新时间。
*   页面支持三类行内操作：修改、重置密码、账号删除。
*   “修改”操作采用弹窗，不再使用角色/单位双下拉即时提交；弹窗内用户名只读，真实姓名、角色、所属单位可编辑。
*   当角色切换为系统管理员时，所属单位自动清空并禁用；当前登录管理员本人账号不可修改自己的角色。
*   “重置密码”采用独立弹窗，只允许系统管理员为其他账号设置新密码。
*   删除按钮使用危险态按钮；当前登录管理员本人账号不可删除，避免误删后失去系统管理入口。
*   删除确认文案需要明确提示：系统会自动清退该用户名下保管设备；若仍有关联未完结检修工单，后端会阻断删除并返回错误提示。

### 4.2 顶栏改密补充规范
*   右上角当前登录用户区域新增“修改密码”文本按钮。
*   点击后弹出“旧密码 / 新密码 / 确认新密码”表单。
*   改密成功后必须清空本地登录态并跳转登录页，要求用户使用新密码重新登录。

## 5. 数据看板 (Dashboard) UI 规范 (新)
*   **排版栅格系统**：使用 `el-row` 和 `el-col`（结合 `span` 或 `:xs`, `:sm`, `:md`, `:lg`）构建自适应网格。指标卡片在 1920px 屏幕下每行 4~5 列，在 1280px 下换行或堆叠。
*   **指标卡片 (KPI Cards)**：采用白色卡片背景，左侧为彩色状态圆角图标，右侧为大字号数字指标（如原值使用货币符号 `￥` 并保留 2 位小数），数字下方为小字号灰色指标名称。
*   **图表卡片**：ECharts 图表需包裹于固定高度（如 `350px`）的 `div` 容器内。
*   **图表交互与自适应**：
    - 图表初始化必须在组件 `mounted` 声明周期进行，并在 `beforeDestroy` 时销毁实例，防内存泄漏。
    - 绑定 `window.addEventListener('resize', this.handleResize)` 以在视口变化时调用 `chart.resize()` 自适应重绘。
*   **待办工作台表格**：看板下方的“待办”或“我的设备”等列表直接复用简洁的 `el-table`，但去除复杂的搜索过滤区，以最直观的信息列配合 “处理” 跳转按钮（使用 `cursor-pointer`）形式展现。

## 6. 数据治理 (Governance) UI 规范 (新)
*   **入口权限**：左侧导航新增“数据治理”或“风险分析”入口，仅 Role 2 资产管理员和 Role 3 系统管理员可见。
*   **页面结构**：
    - 顶部使用 KPI 卡片展示数据质量问题数、高风险设备数、中风险设备数、长期空闲设备数和维修成本异常数。
    - 中部使用图表或进度条展示风险等级分布、部门风险分布和分类风险分布。
    - 下方使用 `el-table` 展示风险设备清单，支持风险等级、单位、分类筛选。
*   **状态色**：
    - 高风险使用危险色 `#F56C6C`。
    - 中风险使用警告色 `#E6A23C`。
    - 低风险使用成功色 `#67C23A` 或信息色。
    - 数据质量问题使用警告标签，并在详情列说明具体原因。
*   **交互状态**：
    - 页面加载时使用 `v-loading` 或骨架状态。
    - 无风险数据时展示空状态，不显示空白页面。
    - 接口失败时展示错误提示和重试入口。
*   **响应式要求**：1280px 宽度下 KPI 卡片自动换行；风险设备表格保持横向信息密度，必要时通过分页和列宽控制避免页面底部横向滚动。

## 7. 事件中心 (Event Center) UI 规范 (新)
*   **入口权限**：左侧导航或顶栏角标可进入“消息中心”或“待处理事件”页；Role 1/2 可见本人待处理事件，Role 3 仅保留只读监控入口。
*   **页面结构**：
    - 顶部展示未读数、事件类型筛选和刷新入口。
    - 中部使用列表或卡片展示事件标题、摘要、来源类型、创建时间和已读状态。
    - 右侧或弹窗展示事件详情，并提供“标记已读”“去处理”操作。
*   **交互状态**：
    - 第一版允许手动刷新或进入页面后自动拉取，不要求 WebSocket 实时推送。
    - 无事件时展示空状态，而不是空白列表。
    - 已失效或已处理事件可使用弱化样式展示，避免与待处理事件混淆。
*   **跳转约束**：
    - 高风险设备事件跳转到治理页或设备详情页。
    - 审批积压事件跳转到领用审批页。
    - 维修超时事件跳转到检修记录页。

## 8. AI 辅助 (AI Assistant) UI 规范 (新)
*   **入口权限**：左侧导航新增“AI 助手”或“AI 报告”入口，仅 Role 2 资产管理员和 Role 3 系统管理员可见。
*   **页面结构**：
    - 顶部展示 AI 功能边界提示，明确“AI 输出仅为草案，需人工确认”。
    - 左侧提供报告类型选择：周报、月报、设备生命周期摘要。
    - 右侧展示生成结果，包括报告正文、风险证据、建议动作和复制按钮。
*   **交互状态**：
    - 生成中使用 `v-loading` 和按钮 loading，防止重复提交。
    - Provider 未配置时展示“AI 服务未配置”的空状态和处理建议。
    - Provider 调用失败时展示错误提示和重试按钮。
*   **安全呈现**：
    - 所有 AI 文本区域必须带“草案”或“建议”标识。
    - 不提供“自动审批”“自动报废”“自动恢复数据库”等高风险按钮。
    - 复制草案只复制文本，不触发任何业务写操作。

## 9. 检修流转闭环与复核 UI 规范 (新)
*   **工单状态标签**：
    - 0-待指派：使用灰色/信息色 (`#909399` 或 `info`)。
    - 1-维修中：使用警告色 (`#E6A23C` 或 `warning`)。
    - 2-待复核：使用主色 (`#409EFF` 或 `primary`)。
    - 3-已复核可用：使用成功色 (`#67C23A` 或 `success`)。
    - 4-转报废：使用危险色 (`#F56C6C` 或 `danger`)。
*   **指派与完工修改弹窗交互（多态弹窗）**：
    - 根据工单所处状态，表单展示和输入被动态划分：
      - 状态为 0（指派）：只允许选择检修人（从维修工下拉列表选择），屏蔽/隐藏检修费用与登记内容输入框。
      - 状态为 1（完工）：只允许输入检修内容和检修费用。检修人和设备编号被禁用只读。
*   **完工复核弹窗交互**：
    - 新增独立复核处置对话框，当资产管理员（Role 2）点击处于“待复核”工单行内的“复核”按钮时唤起。
    - 表单包含：
      - 复核结论：单选框组合（“复核通过恢复可用” / “转报废”）。
      - 复核意见：多行文本输入框，无论哪种结论均推荐填写。
      - 报废原因：多行文本输入框，仅当结论切换为“转报废”时显示，且作为必填项实施前置校验，确保报废有因可查。

## 10. TASK-017 全局 UI/UX 收尾规范 (新)
*   **统一页头模板**：
    - 关键页面统一采用“标题 + 副标题 + 右侧主操作 + 可选摘要条/摘要卡”的信息层级。
    - 标题用于说明页面目标，副标题用于说明当前页面的处理语境或边界，不再只写空泛欢迎语。
    - 右侧主操作保持单一且稳定，优先放刷新、进入处理或新增入口；批量辅助操作放在内容区内。
*   **导航信息架构**：
    - 左侧导航保留“数据看板”“消息中心”一级直达。
    - 资产执行类页面归入统一业务组，基础资料与系统治理分组独立展示。
    - 菜单命名以用户任务为导向，例如“领用审批”“检修记录”“备份与恢复”，避免同层同时混用对象名和技术名。
*   **状态语义映射**：
    - 待处理类状态优先使用 `info / primary / warning` 递进表达，例如待指派、待复核、待审批。
    - 已完成或已通过类状态统一使用 `success`。
    - 风险、拒绝、失败、报废等高警示状态统一使用 `danger`。
    - 同义状态跨页面必须保持同色同文案，不允许同义不同色或同色不同义。
*   **消息与待办联动**：
    - 消息中心是待办入口页，不是纯通知陈列页；“去处理”按钮必须带用户进入真实处理页面。
    - `maintenance` 消息跳转检修页时保留 `maintId`，`claim` 消息跳转领用审批页时保留 `claimId`，目标记录需优先排序、高亮并自动滚动到可视区域。
    - 关键表格页的操作列固定在右侧，避免用户因横向滚动丢失处理按钮。
*   **摘要条与空状态**：
    - 待办型表格页允许使用轻量摘要条作为视图筛选器，默认优先服务“待处理”任务，不替代详细筛选表单。
    - 无未读、无待办、无匹配记录、目标记录失效四类场景都必须展示统一空状态，不得出现整页空白。
    - 目标记录不存在时允许 toast 提示，但还需保留页内说明文案，避免用户只能依赖瞬时消息理解状态。

### 11.1 交互口径补充
- 侧边栏菜单和路由守卫都使用角色判断，但后端仍是最终权限边界。
- 消息中心是拉取式页面，不承诺实时 WebSocket 推送。
- AI 助手页面只展示草案、复制、导出/打印类辅助能力，不提供自动审批、自动报废、自动恢复数据库等高风险按钮。
- 设备“可领用”在 UI 上应解释为“在用且暂无保管人”，避免与数据库不存在的“闲置”状态混用。
