- 审查目标：`docs/4-tasks/features/TASK-019-write-web-course-report/course_design_report_final.docx`
- 对照依据：
  - `docs/4-tasks/features/TASK-018-web-course-report/report_plan.md`

  - `docs/4-tasks/features/TASK-019-write-web-course-report/tasks.md`
  - `docs/2-designs/`
  - 当前前后端代码与截图资源

## 2. 总体结论

结论：`TASK-019` 当前 `.docx` 产物不满足 `TASK-018` 报告写作计划书的完整要求，暂不建议作为最终提交版。

当前产物具备可用基础，正文主体、章节框架、测试章节和大部分文字说明已经形成，但在以下方面仍存在明显缺口：

- 运行效果截图覆盖不完整。
- 图表和流程图未以正式报告可用形式交付。
- Word 目录和标题层级未按正式文档方式落地。
- 部分内容存在与真实项目资料不一致的问题。
- 封面信息仍有未完成项，但该项可由作者自行补齐。

## 3. 审查结果明细

### 3.1 截图覆盖不满足计划书要求

`TASK-018` 的截图清单要求至少覆盖 20 项页面或场景，包括登录页、注册页、数据看板首页、设备新增/编辑弹窗、调拨记录页面、数据备份恢复页面、AI 报告导出或打印效果等。

但 `TASK-019` 正文中实际引用的图片只有 9 张：

- 登录界面
- 设备台账列表
- 领用审批历史
- 检修记录
- 数据看板
- 数据治理
- 消息中心
- 设备详情 AI 摘要弹框
- AI 报告草案

同时，`screenshots/device_transfer.png` 虽然存在于产物目录中，但未被嵌入最终 `.docx`。

审查结论：

- 不满足 `TASK-018` 对“运行效果截图覆盖核心模块”的要求。
- 不满足 `TASK-019` 对截图检查和补齐的完成标准。

### 3.2 图表、流程图、时序图未正式落地

`TASK-018` 明确要求准备以下图示内容：

- 用例图
- 数据流图
- 权限矩阵表
- 至少 4 张核心时序图

但在当前 `TASK-019` 产物中，`graph TD`、`erDiagram`、代码块等内容仍以 Markdown/mermaid 源文本形式存在，导出到 `.docx` 后并未渲染为正式插图，而是落成了表格中的源码文本。

此外，正文中还写到“本小节展示系统两个核心业务时序图的逻辑设计”，与 `TASK-018` 计划书要求的“至少 4 张”不一致。

审查结论：

- 权限矩阵表已具备表格形式，可视为部分满足。
- 用例图、数据流图、E-R 图、时序图未达到正式报告可提交标准。
- 该项整体判定为不通过。

### 3.3 Word 目录与标题层级不满足正式排版要求

`TASK-019` 任务要求包含以下检查：

- 标题层级
- 目录生成条件
- 图表编号
- 截图说明

经检查，最终 `.docx` 存在以下问题：

- 文档中没有 TOC 字段。
- 章节标题未使用 Word 的标准标题样式。
- 目录是手工项目符号列表，不是可自动更新目录。
- 图片缺少统一图号，如“图 5-1”“图 5-2”。
- 图片说明存在，但不构成规范图注体系。

审查结论：

- 不满足正式课程设计报告对目录与层级可维护性的要求。
- 不满足 `TASK-019` 中“目录生成条件、标题层级、图表编号”的完成标准。

### 3.4 部分内容与项目真实资料不一致

正文中存在若干与项目文档、数据库设计或代码实现不一致的内容，例如：

1. 单位字段描述不一致
   - 报告中多处使用 `dept_id`、`from_dept_id`、`to_dept_id`、`BaseContext.getDeptId()`
   - 但当前项目设计与数据库文档实际使用的是 `unit_code`、`out_unit_code`、`in_unit_code`、`BaseContext.getCurrentUnitCode()`

2. 数据结构表达偏离真实项目
   - 附录中的若干表结构使用了更偏抽象化或二次整理后的字段命名，不完全对应 `docs/2-designs/db_schema.md` 中的真实字段定义

3. 时序图与设计描述存在“应有内容”和“实际交付内容”之间的不一致
   - 文字说明写了图示设计，但产物中没有形成对应正式图件

审查结论：

- 不满足 `TASK-019` 关于“内容必须可追溯、可核验、不得编造或偏离项目资料”的要求。
- 该项属于真实性问题，应优先修正。

### 3.5 封面信息未完成

封面仍保留作者自行填写项，属于未最终落版状态。该问题主要包括：

- 题目位置未完整落入最终封面字段
- 姓名、学号未填写
- 专业班级仍保留占位表达

说明：

- 该项可由作者自行补写，因此不作为本次后续修复建议中的重点技术问题。
- 但若按“最终提交版”标准审查，该项仍应视为未完成。

## 4. 通过项

以下内容当前可以判定为通过或基本通过：

- 文档已形成完整章节框架，覆盖摘要、正文、测试、总结、参考文献、附录等部分。
- 参考文献数量达到 10 篇，满足最低数量要求。
- 未发现密码、Token、API Key 等明显敏感信息泄露。
- 截图整体使用测试或演示数据，未见高风险敏感展示。
- 实际执行验证命令后，后端 `mvn test` 通过，前端 `npm run lint` 通过。

## 5. 风险说明

除上述主要问题外，还存在以下补充风险：

- `第 7 章 课程设计总结` 内容篇幅偏长，如教师严格按模板执行“总结不超过 500 字”，则可能超限。
- `docx` 当前对 Markdown 内容的落版偏机械，代码块、mermaid、目录、图注均缺少正式论文式整理。
- 当前文档更接近“内容完整的转换稿”，尚未达到“排版完成的正式课程设计报告”。

## 6. 最终判定

最终判定：**不通过**

判定原因如下：

- 截图覆盖未达到 `TASK-018` 计划书要求。
- 用例图、数据流图、E-R 图、时序图未以正式图件交付。
- 目录、标题样式、图号体系未满足正式 Word 报告要求。
- 部分字段、单位隔离表达与真实项目资料不一致。

建议将当前产物视为“可继续修订的中间稿”，在补齐截图、修正图件交付方式、统一 Word 结构化排版、核对字段真实性之后，再作为最终提交版本使用。
